# 工作流节点模型选择Bug修复 方案

## 一、完整调用链路

```
工作流引擎触发节点执行
  ↓
McpToolNode.onProcess()         →  config.getModelName() → "gj-deepseek"（节点配置默认值）
LLMAnswerNode.onProcess()       →  config.getModelName() → 节点配置的模型名称
ClassifierNode.onProcess()      →  config.getModelName() → 节点配置的模型名称
FaqExtractorNode.onProcess()    →  config.getModelName() → 节点配置的模型名称
KeywordExtractorNode.onProcess()→  config.getModelName() → 节点配置的模型名称
  ↓
WorkflowUtil.streamingInvokeLLM(wfState, nodeState, node, modelName, prompt, silentMode)
WorkflowUtil.invokeLLM(wfState, modelName, prompt)
  ↓
[Bug在此处] 第168行/第92行：
  AiModelConfigVo modelConfig = aiChatBaseService.getModule(request, null);
  → modelName 参数完全被忽略
  → 始终调用 getDefaultModelConfigWithKey("LLM")
  → 始终从 ai_config 表取 DEFAULT_LLM_MODEL_ID 对应的模型
  ↓
模型实际使用 DEFAULT_LLM_MODEL_ID=10（闲鱼-星火api-deepseek-v3）
但日志显示的是节点配置的 modelName="gj-deepseek"（误导）
```

## 二、问题根因分析

### 数据库现状
| 字段 | 值 |
|------|-----|
| ai_model_config.name | 显示名称（如 "gj-deepseek"、"闲鱼-星火api-deepseek-v3"） |
| ai_model_config.model_name | 实际模型标识符（如 "Qwen/Qwen3-32B"、"deepseek-v3"） |
| DEFAULT_LLM_MODEL_ID | 10 → 闲鱼-星火api-deepseek-v3 |

### 节点存储的 model_name 字段
- 节点配置存储的是 `ai_model_config.name`（显示名称）
- `selectByModelName()` 的 SQL：`WHERE t1.name = #{modelName} AND t1.enabled = 1`
- 因此节点配置 → `getModelConfigByName(modelName)` 查询路径完全正确，只是未被调用

### Bug点
**`WorkflowUtil.java`**：
- `invokeLLM()` 第92行：忽略 `modelName` 参数
- `streamingInvokeLLM()` 第168行：忽略 `modelName` 参数
- 两个方法都日志记录了 `modelName` 但实际不使用，造成误导

## 三、按文件的修改设计

### 后端

#### 修改文件：`WorkflowUtil.java`
路径：`scm-ai/src/main/java/com/xinyirun/scm/ai/workflow/WorkflowUtil.java`

**修改1：`invokeLLM()` 方法（第92行）**

原代码：
```java
var modelConfig = aiChatBaseService.getModule(request, null);
```

改为：
```java
AiModelConfigService aiModelConfigService = SpringUtil.getBean(AiModelConfigService.class);
AiModelConfigVo modelConfig = resolveModelConfig(modelName, aiModelConfigService, aiChatBaseService, request);
```

**修改2：`streamingInvokeLLM()` 方法（第168行）**

原代码：
```java
AiModelConfigVo modelConfig = aiChatBaseService.getModule(request, null);
```

改为：
```java
AiModelConfigService aiModelConfigService = SpringUtil.getBean(AiModelConfigService.class);
AiModelConfigVo modelConfig = resolveModelConfig(modelName, aiModelConfigService, aiChatBaseService, request);
```

**新增私有方法 `resolveModelConfig()`**：

```java
private static AiModelConfigVo resolveModelConfig(String modelName,
                                                    AiModelConfigService aiModelConfigService,
                                                    AiChatBaseService aiChatBaseService,
                                                    AIChatRequestVo request) {
    if (StringUtils.isNotBlank(modelName)) {
        try {
            return aiModelConfigService.getModelConfigByName(modelName);
        } catch (Exception e) {
            log.warn("指定模型 [{}] 未找到，使用系统默认模型", modelName);
        }
    }
    return aiChatBaseService.getModule(request, null);
}
```

**修改3：修正日志内容（在获取到 modelConfig 之后打印实际使用的模型）**

在两个方法中，将日志由打印参数 `modelName` 改为打印实际使用的 `modelConfig.getName()`。

### 前端
无需修改。

### 数据库
无需修改。

## 四、KISS原则7问题

1. **这是个真问题还是臆想出来的？** 真问题。日志记录证实，节点模型配置完全无效。
2. **有更简单的方法吗？** 是的，抽取一个9行的私有方法解决，修改2处调用点。
3. **会破坏什么吗？** 不会。行为是渐进式改进：若节点指定了有效模型名则用之，否则降级到默认模型，和之前行为一致。
4. **当前项目真的需要这个功能吗？** 是的。节点级模型选择是工作流的核心设计能力。
5. **过度设计了吗？** 没有。只改2行，加1个方法，总共约13行新代码。
6. **话题是否模糊？** 不模糊。代码已读清楚，问题定位精确。
7. **是否学习了代码实施的注意事项？** 是。无需QueryWrapper（无DB写操作），只调用已有Service方法。
