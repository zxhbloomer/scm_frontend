# scm-ai 模块规范整改方案

**日期**: 2026-03-07 15:00:00
**需求来源**: 代码规范整改 + 知识库下拉排除临时知识库 Bug 修复

---

## 一、问题背景

scm-ai 模块存在两类规范违反问题：

1. **Bug**: 知识库检索节点属性面板的下拉选项中，显示了临时知识库（`is_temp=1`），应排除
2. **规范违反**: 多个 Service 文件使用 `LambdaQueryWrapper`/`QueryWrapper` 进行查询，违反 SCM 规范第17、24条（查询必须使用 SQL 写在 Mapper 中）

---

## 二、调用链路分析

### Bug 修复链路（知识库下拉排除临时知识库）

```
前端 KnowledgeRetrievalNodeProperty.vue
  → loadPermanentKnowledgeBases()
  → import knowledgeBaseSearchMine / knowledgeBaseSearchPublic from utils
  → GET /ai/knowledge-base/search/mine?keyword=&page=1&pageSize=100
  → GET /ai/knowledge-base/search/public?keyword=&page=1&pageSize=100
  → KnowledgeBaseController.searchMine() / searchPublic()
  → KnowledgeBaseService.searchMine() / searchPublic()
  → AiKnowledgeBaseMapper.searchMineExcludeTemp() / searchPublicExcludeTemp()  ← 已修改
```

### QueryWrapper 整改链路

各 Service 直接调用 `mapper.selectOne(new LambdaQueryWrapper<>().eq(...))` → 需改为 mapper 中定义 SQL 方法，Service 调用 SQL 方法。

### 3.4 前端问题

**camelCase 字段引用结论**：前端 camelCase 字段引用是正确的，不需要修改。

原因：scm-ai 配置 `map-underscore-to-camel-case: false`，所以 SQL 中用了 `AS kbUuid`、`AS isPublic` 等驼峰别名，后端返回给前端的 JSON 字段就是 camelCase。前端引用 `item.kbUuid`、`item.isPublic` 等是正确的。

**isTemp 类型不匹配 Bug（需修复）**：

经过前后端 VO 字段比对，发现 `AiKnowledgeBaseVo.isTemp` 字段类型为 `Boolean`，后端 JSON 序列化后返回 `true`/`false`。但前端多处使用 `=== 1` 进行比较，导致判断永远为 false（临时知识库的操作限制失效）。

| 文件 | 行号 | 问题代码 | 修复代码 |
|------|------|---------|---------|
| `KnowledgeBaseManageDialog.vue` | 139 | `:disabled="scope.row.isTemp === 1"` | `:disabled="!!scope.row.isTemp"` |
| `KnowledgeBaseDetailDialog.vue` | 28 | `:disabled="!kbUuid \|\| kbInfo.isTemp === 1"` | `:disabled="!kbUuid \|\| !!kbInfo.isTemp"` |
| `KnowledgeBaseDetailDialog.vue` | 36 | `:disabled="!kbUuid \|\| kbInfo.isTemp === 1"` | `:disabled="!kbUuid \|\| !!kbInfo.isTemp"` |
| `KnowledgeBaseDetailDialog.vue` | 44 | `:disabled="selectedItems.length === 0 \|\| kbInfo.isTemp === 1"` | `:disabled="selectedItems.length === 0 \|\| !!kbInfo.isTemp"` |
| `KnowledgeBaseDetailDialog.vue` | 215 | `v-if="kbInfo.isTemp !== 1"` | `v-if="!kbInfo.isTemp"` |
| `KnowledgeBaseDetailDialog.vue` | 385 | `kbInfo: { isTemp: 0 }` | `kbInfo: { isTemp: false }` |
| `KnowledgeBaseDetailDialog.vue` | 441 | `this.kbInfo = kbResponse.data \|\| { isTemp: 0 }` | `this.kbInfo = kbResponse.data \|\| { isTemp: false }` |
| `KnowledgeBaseDetailDialog.vue` | 444 | `this.kbInfo = { isTemp: 0 }` | `this.kbInfo = { isTemp: false }` |

**KnowledgeRetrievalNodeProperty.vue**：`loadPermanentKnowledgeBases()` 方法已通过后端 SQL 排除临时知识库，前端无需修改。

## 四、KISS 原则 7 问回答

1. **真问题还是臆想？** ✅ 真问题。QueryWrapper 违反 SCM 规范，且知识库下拉显示临时知识库是真实 Bug。
2. **有更简单的方法吗？** ✅ 最简方案：在 Mapper 中写 SQL，Service 直接调用，不引入新的抽象层。
3. **会破坏什么吗？** ✅ 不会。SQL 逻辑与 QueryWrapper 等价，只是换了写法。
4. **当前项目真的需要这个功能吗？** ✅ 必须做，规范整改是代码质量基础。
5. **过度设计了吗？** ✅ 没有。每个 QueryWrapper 对应一个 SQL 方法，一对一替换。
6. **话题是否模糊？** ✅ 清晰。每个文件的修改内容明确。
7. **是否学习了代码实施注意事项？** ✅ 是。遵循规范17、24：查询用 SQL 写在 Mapper，插入/更新用 bean 操作。
