# 知识库检索节点支持多知识库查询 - 开发设计方案

**日期**: 2026-03-07
**功能**: 知识库检索节点（KnowledgeRetrievalNode）支持选择多个知识库并行查询

---

## 一、完整调用链路

```
用户在属性面板选择多个知识库
    ↓
KnowledgeRetrievalNodeProperty.vue（前端属性面板）
    - nodeConfig.knowledge_base_uuids: [] 数组
    - nodeConfig.knowledge_base_list: [{uuid, name, is_temp, temp_node_uuid}]
    ↓
workflowApi.js → POST /api/v1/ai/workflow/update（保存工作流）
    - nodeConfig 原样序列化到后端
    ↓
后端 AiWorkflowNodeVo.nodeConfig（JSON存储）
    ↓
工作流执行：POST /api/v1/ai/workflow/run/{uuid}（SSE）
    ↓
KnowledgeRetrievalNode.onProcess()
    ↓
KnowledgeRetrievalNodeConfig 反序列化
    - knowledgeBaseUuids: List<String>（新增）
    - knowledgeBaseUuid: String（保留，向后兼容）
    ↓
[并行] CompletableFuture × N个知识库
    ├→ MilvusVectorRetrievalService.searchSimilarDocuments(text, kbUuid1, topN, minScore)
    ├→ MilvusVectorRetrievalService.searchSimilarDocuments(text, kbUuid2, topN, minScore)
    └→ MilvusVectorRetrievalService.searchSimilarDocuments(text, kbUuidN, topN, minScore)
    ↓
合并所有结果 → 按 score 降序 → 过滤阈值 → 限制 topN
    ↓
[可选] 图谱检索（对每个知识库并行执行）
    ↓
拼接结果文本 → NodeProcessResult
    ↓
SSE 流式输出到前端
```

---

## 二、问题诊断与根因

**现状**：`KnowledgeRetrievalNodeConfig.knowledgeBaseUuid` 是单个 String，前端 `el-select` 无 `multiple`。

**用户变通方案**：放多个检索节点 + 合并结果节点，画布复杂，连线多。

**根因**：设计之初只考虑了单知识库场景，数据结构是单值而非数组。

---

1. **真问题**：✅ 用户截图已证明在生产中用多节点变通

## 四、按文件设计

### 后端（2个文件）

#### 4.1 KnowledgeRetrievalNodeConfig.java
**路径**: `scm-ai/src/main/java/com/xinyirun/scm/ai/workflow/node/knowledgeretrieval/KnowledgeRetrievalNodeConfig.java`

**改动**：新增两个字段，保留旧字段

```java
// 新增：多知识库UUID列表
@JsonProperty("knowledge_base_uuids")
private List<String> knowledgeBaseUuids;

// 新增：多知识库信息列表（含名称，用于前端显示）
@JsonProperty("knowledge_base_list")
private List<KbItem> knowledgeBaseList;

// 内部类
@Data
public static class KbItem {
    @JsonProperty("uuid")
    private String uuid;
    @JsonProperty("name")
    private String name;
    @JsonProperty("is_temp")
    private Boolean isTemp;
    @JsonProperty("temp_node_uuid")
    private String tempNodeUuid;
}
```

旧字段 `knowledgeBaseUuid` / `knowledgeBaseName` / `isTempKb` / `tempKbNodeUuid` **全部保留**，不删除。

**兼容逻辑**（在 Node 层处理）：
- 若 `knowledgeBaseUuids` 非空 → 使用多知识库模式
- 若 `knowledgeBaseUuids` 为空但 `knowledgeBaseUuid` 非空 → 兼容旧单知识库模式，自动转为单元素列表

---

#### 4.2 KnowledgeRetrievalNode.java
**路径**: `scm-ai/src/main/java/com/xinyirun/scm/ai/workflow/node/knowledgeretrieval/KnowledgeRetrievalNode.java`

**改动**：`onProcess()` 方法重构，核心逻辑：

```
1. 获取知识库UUID列表（兼容新旧格式）
   - 新格式：knowledgeBaseUuids 列表
   - 旧格式：knowledgeBaseUuid 单值 → 包装成列表
   - 临时知识库：从 state.data() 解析 kbUuid，加入列表

2. 获取查询关键词（逻辑不变）

3. 并行向量检索（新增并行逻辑）
   - 为每个 kbUuid 创建 CompletableFuture
   - 调用现有 searchSimilarDocuments()
   - 合并结果 → 按 score 降序 → 过滤阈值 → 限制 topN

4. 并行图谱检索（若启用）
   - 同样并行处理每个 kbUuid
   - 合并图谱结果

5. 拼接输出（逻辑不变）
```

**不改动**：`MilvusVectorRetrievalService`、`GraphRetrievalService`，直接复用现有方法。

## 五、向后兼容策略

| 场景 | 处理方式 |
|------|---------|
| 旧工作流（单知识库） | 后端：`knowledgeBaseUuids` 为 null，读 `knowledgeBaseUuid` 包装成列表 |
| 旧工作流（单知识库） | 前端：`initializeSelection` 检测旧字段，转为数组显示 |
| 新工作流（多知识库） | 前端写 `knowledge_base_uuids` 数组，后端直接读取 |
| 临时知识库 | 支持混合：永久+临时可同时选择 |

---

## 六、风险分析

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 旧工作流不兼容 | 低 | 保留旧字段，兼容逻辑在 Node 层处理 |
| 并行检索超时 | 低 | 复用现有超时机制，单个失败不影响其他 |
| 多知识库结果过多 | 低 | topN 限制总结果数，不是每个知识库 topN |
| 临时知识库混合 | 低 | 临时知识库解析逻辑已有，扩展到列表即可 |
