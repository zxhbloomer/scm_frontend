# 第三方供应商模型接入方案

**日期**: 2026-03-01
**版本**: v1.0
**范围**: AI模型设置 - 新增"第三方"供应商入口，支持动态获取模型列表

---

## 一、调用链路分析

### 获取远程模型列表：
```
用户点"获取模型"
→ 前端 api/model.js fetchRemoteModels({baseUrl, apiKey})
→ POST /api/v1/ai/model/config/fetch-remote-models
→ AiModelConfigController.fetchRemoteModels()
→ AiModelConfigService.fetchRemoteModels()
→ RestTemplate GET {baseUrl}/v1/models (Authorization: Bearer apiKey)
→ 解析OpenAI兼容响应 → 返回模型ID列表
→ 前端填充基础模型下拉框
```

### 保存模型（复用现有链路）：
```
用户选模型后点保存 → editModelConfig() → 现有保存逻辑（不变）
provider字段值为 'ThirdParty'，其余逻辑与现有供应商完全一致
```

## 二、方案设计（按文件）

### 前端修改

#### 文件1：`constants/model.js`
- `ModelBaseTypeEnum` 新增 `ThirdParty: 'ThirdParty'`
- `modelList` 数组新增第三方供应商项（icon: 'thirdPartyApi'）
- `baseModelTypeMap` 新增 `[ThirdParty]: []`（空数组，运行时动态填充）
- `defaultAdvancedSetValueMap` 新增 `[ThirdParty]` 配置（通用默认值）

#### 文件2：`api/model.js`
- 新增 `fetchRemoteModels(data)` 函数
  - POST /api/v1/ai/model/config/fetch-remote-models
  - 参数：{ baseUrl, apiKey }
  - 返回：模型ID列表

#### 文件3：`ModelAddDialog.vue`（复用，加条件分支）

核心改动：
- `data()` 新增 `remoteModelOptions: []` 和 `fetchingModels: false`
- `computed` 新增 `isThirdParty`：判断 `supplierModelItem.value === 'ThirdParty'`
- `template`：
  - 基础模型的 el-autocomplete 旁加"获取模型"按钮（v-if="isThirdParty"）
  - 按钮仅在 baseUrl 和 apiKey 都填写后可点击
- `methods`：
  - 新增 `fetchRemoteModelList()` 方法：
    1. 校验 baseUrl 和 apiKey 不为空
    2. 调用 `fetchRemoteModels({baseUrl, apiKey})`
    3. 成功：填充 `remoteModelOptions`，提示"获取到N个模型"
    4. 失败：提示"连接失败: 错误信息"
  - 修改 `fetchSuggestions()`：isThirdParty 时从 `remoteModelOptions` 取数据
  - 修改 `selectAutoComplete()`：isThirdParty 时，能力标记默认按 modelType 推导

#### 文件4：`utils/modelUtils.js`
- `getModelSvg()` 新增 ThirdParty case，返回 'thirdPartyApi'

#### 文件5：`src/icons/svg/thirdPartyApi.svg`（新建）
- 简洁的通用API图标（插头/连接器形状）

### 后端修改

#### 文件6：`AiModelConfigController.java`
- 新增接口：`POST /fetch-remote-models`
- 接收 `FetchRemoteModelsVo`（含 baseUrl, apiKey）
- 调用 Service 层获取远程模型列表

#### 文件7：`AiModelConfigService.java`
- 新增方法：`fetchRemoteModels(String baseUrl, String apiKey)`
  - 使用 RestTemplate 调用 `{baseUrl}/v1/models`
  - 设置 Authorization: Bearer {apiKey} 请求头
  - 解析响应 JSON 的 data[].id 字段
  - 返回模型ID字符串列表
  - 超时时间：10秒
  - 异常处理：连接超时、认证失败、格式错误

#### 文件8：新建 `FetchRemoteModelsVo.java`
- 位置：scm-ai/bean/vo/config/
- 字段：baseUrl（String）、apiKey（String）
- 用于接收前端获取远程模型列表的请求参数

## 三、KISS原则评估

1. **真问题？** ✅ 用户需要接入预定义之外的AI服务
2. **更简方案？** ✅ 复用现有弹窗，只加一个条件分支 + 一个按钮
3. **会破坏？** ✅ 不会，新增供应商选项，现有逻辑零影响
4. **真需要？** ✅ 企业AI系统必须支持自定义接入
5. **过度设计？** ✅ 没有，一个按钮做两件事（测试+获取模型）
6. **幻觉？** ✅ 无，基于OpenAI兼容协议标准
7. **注意事项？** ✅ 后端用RestTemplate（已有），bean操作不涉及（无DB新增表）

## 五、风险分析

| 风险 | 影响 | 缓解 |
|------|------|------|
| 第三方API不兼容OpenAI协议 | 获取模型列表失败 | 前端显示错误提示，用户可手动输入模型名 |
| 第三方API响应慢 | 用户等待时间长 | 后端设10秒超时，前端显示loading状态 |
| CORS问题 | 直接从前端调会被拦截 | 通过后端代理调用，无CORS问题 |
